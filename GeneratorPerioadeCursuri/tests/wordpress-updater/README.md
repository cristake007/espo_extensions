# WordPress updater contract fixtures

These fixtures lock Phase 0 of the WordPress course updater port. They are test
inputs and expected contracts only; they do not implement the updater and they
must never be used to contact a live WordPress site.

## Pinned source

- Repository: `cristake007/platforma`
- Commit: `ce7caf4196c107625ba8d553523bca79c9f13f8a`
- Upstream test file: `apps/planificator/tests_course_updater.py`
- Upstream implementation: `apps/planificator/wp_course_updater.py`

`upstream-assertions.json` maps every active test in the pinned upstream test
file to a planned PHP assertion. Future phases should keep those assertion names
or update this map deliberately when the PHP test classes are introduced.

## Fixture inventory

- `fixtures/schedules/romanian-nume-curs.csv`: current Espo export title and
  Romanian month headers.
- `fixtures/schedules/django-title-english.csv`: pinned Django `Title` input and
  English month headers.
- `fixtures/schedules/bilingual-precedence.csv`: English-first month precedence
  with Romanian fallback.
- `fixtures/schedules/semicolon-bom.csv`: UTF-8 BOM and semicolon delimiter.
- `fixtures/schedules/edge-rows.csv`: empty row, empty title, missing/invalid
  permalink, no dates, and textual `nan`.
- `fixtures/schedules/typed-date.xlsx`: a real numeric XLSX date cell with a
  date number format, generated by `generate-typed-date-fixture.php`.
- `fixtures/expected/preview.json`: ordered preview expectations for all schedule
  fixtures.
- `fixtures/expected/merge.json`: fixed-clock date parsing, filtering,
  deduplication, ordering, payload, and no-change expectations.
- `fixtures/http/scenarios.json`: DNS and mocked transport exchanges for URL,
  redirect, retry, response-bound, challenge, JSON, and timeout behavior.

The reserved `.test` hostnames are documentation-only inputs. HTTP scenarios
contain synthetic DNS answers and transport results; automated checks must inject
them and must not resolve or request their hosts.

## Intentional Espo compatibility and security differences

These adaptations are authoritative for the PHP port and intentionally differ
from a literal copy of the pinned Django behavior:

1. Accept `Nume curs` and `Course Name` in addition to Django `Title`.
2. Enforce 20 MiB, 50-column, and 5,000-non-empty-row boundaries consistently.
3. Keep a date range as one exact ACF value; the upstream `expand_date_token`
   helper is inactive in this workflow and is not ported as updater behavior.
4. Treat invalid schedule date text as a row error instead of expanding or
   normalizing it; ISO datetime text is not an accepted ACF program value.
5. Read only the saved record's native attachment and address rows by the
   server-generated source row. Browser slugs, post IDs, dates, and attachment
   IDs are never authoritative.
6. Store only the normalized base URL and username. The application password is
   transient and absent from fixtures, entities, responses, logs, and storage.
7. Resolve every DNS result and pin an approved address for each request.
   Mixed public/private answers are rejected.
8. Revalidate every redirect. Authenticated redirects are limited to the same
   hostname and port, may upgrade HTTP to HTTPS, and may not downgrade HTTPS or
   forward credentials cross-host.
9. Fetch and update resolve the exact slug again; a returned post ID is display
   only. The unused upstream fallback post-ID behavior is not ported.
10. Preview is local-only. Updates are one row at a time, re-read WordPress
    immediately before comparison, and skip POST when ordered values are equal.

## Validation

Run the lightweight Phase 0 integrity check:

```bash
php GeneratorPerioadeCursuri/tests/wordpress-updater/contract.php
```

Run the Phase 1 parser and merge-policy checks:

```bash
php GeneratorPerioadeCursuri/tests/wordpress-updater/phase-1.php
```

Run the Phase 2 URL, transport, and WordPress protocol checks:

```bash
php GeneratorPerioadeCursuri/tests/wordpress-updater/phase-2.php
```

The Phase 2 runner injects DNS answers, transport responses, time, sleeping, and
jitter. It executes every scenario in `fixtures/http/scenarios.json` without
performing DNS resolution or an HTTP request.

Run the Phase 3 entity, ACL metadata, layout, menu, translation, and native
upload-shell checks:

```bash
php GeneratorPerioadeCursuri/tests/wordpress-updater/phase-3.php
```

This static check does not install the extension or rebuild an EspoCRM instance.
Role ACL behavior, schema rebuild, and browser interaction remain non-production
runtime acceptance checks.

Run the Phase 4 service orchestration, ACL/attachment boundary, strict request,
stale preview, per-row update, safe logging, and route checks:

```bash
php GeneratorPerioadeCursuri/tests/wordpress-updater/phase-4.php
```

The Phase 4 runner replaces Espo persistence/ACL/file storage and the WordPress
client with in-memory test doubles. It does not perform DNS or HTTP operations.

Run the Phase 5 detail-workspace, transient-secret, request-boundary,
accessibility, localization, and scoped-theme checks:

```bash
php GeneratorPerioadeCursuri/tests/wordpress-updater/phase-5.php
```

This is a static client contract check. It does not open a browser or contact
WordPress; interactive browser acceptance remains part of Phase 6 staging.

When the EspoCRM vendor tree is unavailable, the Phase 1 runner loads
`phpspreadsheet-test-double.php`. The double exists only to exercise the locked
XLSX fixture through the production parser's PhpSpreadsheet calls; installed
extensions use EspoCRM's real PhpSpreadsheet classes.

Regenerate the XLSX fixture only through its deterministic generator:

```bash
php GeneratorPerioadeCursuri/tests/wordpress-updater/generate-typed-date-fixture.php
```

The contract check performs no DNS lookup and no HTTP request.
